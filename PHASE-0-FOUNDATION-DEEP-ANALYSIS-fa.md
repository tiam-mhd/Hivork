# Hivork — تحلیل عمیق فاز 0

## 1. خلاصه اجرایی

Hivork یک پلتفرم SaaS ماژولار برای خرده‌فروشی ایران است و ماژول اول آن مدیریت اقساط است. این پروژه به صورت یک مونولیت ماژولار با معماری تمیز، مرزبندی مستحکم تنانسی و انضباط پیاده‌سازی enterprise-grade طراحی شده است.

این تحلیل شامل بررسی فاز 0، تناسب محصول با بازار، معماری، قوانین توسعه و ریسک‌های اجرایی است. تحلیل فقط بر اساس مستندات مخزن انجام شده و هیچ کدی تغییر نکرده است.

## 2. محصول و تناسب بازار

### 2.1 تعریف محصول

- محصول اصلی: مدیریت اقساط برای فروشندگان.
- ارزش اصلی کسب‌وکار: کاهش مدیریت دستی، بهبود وصول، و دید مالی بهتر برای فروشنده.
- ارزش مشتری نهایی: دریافت یادآور از طریق ربات، پورتال PWA، و مشاهده تقویم اقساط.
- دامنه MVP به صورت هوشمندانه محدود شده و شامل درگاه پرداخت آنلاین، حسابداری کامل، اپ native و AI/chatbot نیست.

### 2.2 دلیل انتخاب بازار

- فرهنگ خرید قسطی در ایران در صنایع موبایل، پوشاک، لوازم خانگی، و خدمات قوی است.
- بسیاری از فروشگاه‌های کوچک هنوز با Excel، دفترچه یا چت‌های غیررسمی کار می‌کنند.
- استراتژی مستند شده انتخاب مدیریت اقساط را بر منوی دیجیتال ترجیح داده است، زیرا درد بازار شدیدتر، رقابت کمتر و مسیر درآمد سریع‌تر دارد.

### 2.3 مشتری ایده‌آل

- بهترین هدف: فروشگاه‌های خرده‌فروشی ۲–۱۰ نفره که فروش اقساطی فعال دارند.
- هدف فعلی نیست: سازمان‌های بزرگ و کسب‌وکارهایی که اصلاً فروش اقساطی ندارند.
- اهرم رشد: هر tenant می‌تواند ۵۰–۵۰۰ مشتری را از طریق onboarding با ربات وارد پلتفرم کند.

### 2.4 مسیر ورود به بازار

- روز ۰–۳۰: MVP عملیاتی با ثبت فروش، لیست اقساط، یادآوری خودکار و پنل فروشنده.
- روز ۳۱–۶۰: پیاده‌سازی با ۱۰ فروشنده pilot، مشاهده workflow واقعی و import Excel.
- روز ۶۱–۹۰: تبدیل چند pilot به مشتری پولی و ساخت case study واقعی.

### 2.5 مدل درآمد

- مشتری پرداخت‌کننده: فروشنده (tenant) ماهانه اشتراک می‌پردازد.
- مشتری نهایی (Global Customer) رایگان است تا رشد تسهیل شود.
- بازه قیمت پیشنهادی: ۱۹۹ تا ۴۹۹ هزار تومان ماهانه.

## 3. معماری و استراتژی مهندسی

### 3.1 موضع معماری

- مونولیت ماژولار به همراه کارگرها: `api`, `bot-gateway`, `scheduler`.
- معماری تمیز با جداسازی واضح: `packages/domain`, `packages/application`, `packages/infrastructure`, `apps/*`.
- از روز اول multi-tenant با shared database و `tenant_id` روی همه جداول tenant-scoped.
- طراحی داخلی event-driven با الگوی outbox و workerهای idempotent.

### 3.2 فناوری‌ها

- بک‌اند: NestJS، Prisma، PostgreSQL، Redis، BullMQ.
- فرانت‌اند: Next.js + shadcn/ui.
- ربات: grammY و Bale adapter برنامه‌ریزی شده.
- مونورپو: pnpm + Turborepo.

### 3.3 قوت‌ها

- انضباط معماری قوی: separation of concerns و قوانین صریح از coupling زودرس جلوگیری می‌کند.
- مدل تنانسی و RBAC دقیق باعث کاهش ریسک امنیتی بعدی می‌شود.
- سیاست soft delete حفظ داده‌های مالی/قانونی را تقویت می‌کند.
- قرارداد محور بودن با Zod promises باعث همگونی API و UI می‌شود.
- CI شامل چک hard-delete، اعتبارسنجی Prisma، lint، typecheck، تست و build است.

### 3.4 ریسک‌ها

- هر خطای فیلتر tenant می‌تواند به نشت داده منجر شود.
- منطق session شعبه فعال و data scope پیچیدگی بالایی دارد و باید دقیق اجرا شود.
- ماژول اول حجم حساب‌شده‌ای از قوانین دامنه دارد (پرداخت، اقساط، یادآوری)؛ نیاز به انضباط بالا دارد.
- وابستگی به کانال‌های ربات و یادآور به پایداری عملیاتی نیاز دارد.

## 4. تحلیل فاز 0 بنیاد

### 4.1 دامنه فاز 0

فاز 0 بر بنیان پلتفرم تمرکز دارد، نه روی کامل بودن ویژگی‌ها. شامل موارد زیر است:

- scaffold مونورپو، Docker، تنظیمات مشترک.
- اسکلت اپ‌ها: `api`, `web`, `bot-gateway`, `scheduler`.
- ساختار پکیج‌ها برای core، domain، application، infrastructure، contracts، ui، config، i18n.
- اسکیما DB برای tenant، role، customer، audit، settings، outbox.
- موجودیت‌های اصلی دامنه و RBAC.
- flows احراز هویت، middleware، guardها، soft delete، tenant context.
- قراردادها برای auth و اشیاء اصلی.
- یک vertical slice برای ثبت tenant، ساخت customer و بازیابی.

### 4.2 کیفیت و پوشش تسک‌های فاز 0

- فاز 0 شامل ۵۹ تسک است که در حوزه‌های infrastructure، apps، packages، database، domain core، auth، guards، services، contracts و vertical slice تقسیم شده‌اند.
- README فاز به صراحت self-review و هدف کیفیت ۹۵/۱۰۰ را مشخص کرده است.
- تسک‌های بنیادین شامل قابلیت‌های حیاتی پلتفرم مانند soft delete، ثبت tenant، audit logging و scaffolding use case هستند.

### 4.3 ریسک‌های بنیادین

- تاکید قوی بر مستندسازی و سیاست‌ها خوب است، اما اجرای آن نیازمند هماهنگی دقیق است.
- پلتفرم فقط زمانی موفق می‌شود که `packages/domain` پاک بماند و `packages/contracts` واقعاً بین بک‌اند و فرانت‌اند مشترک استفاده شود.
- داده seed و مایگریشن اولیه اهمیت بالایی دارد چون پلتفرم از نقش‌ها، پلان‌ها و module entitlement از روز اول استفاده می‌کند.

## 5. فرآیند توسعه و قوانین

### 5.1 قوانین کیفیت مستندات

- `DOCUMENTATION_AUTHORING_RULES.md` و `PHASE_EPIC_TASK_AUTHORING_RULES.md` کیفیت تسک و مستندات را بسیار سختگیرانه می‌کنند.
- ساختار فاز و تسک بسیار مشخص است: هر تسک یک فایل، README Epic اجباری، حداقل امتیاز بازبینی.

### 5.2 قوانین مهندسی

- `DEVELOPMENT_RULES.md` سیاست‌های سختگیرانه اجرا را تعریف می‌کند: کنترلرهای نازک، منطق دامنه در use case/entity، API-first، event-driven، filter تنانسی، RBAC guard، audit، Zod contracts.
- `EXCELLENCE-STANDARDS.md` استاندارد enterprise-level را اضافه می‌کند: حالات کامل صفحه، فرم‌های جامع، فیلدهای پایه DB، الگوهای query، مدیریت خطا.
- `SOFT-DELETE-POLICY.md` حذف سخت را روی داده‌های تجاری ممنوع می‌کند و رفتار append-only را برای audit/outbox الزامی می‌سازد.

### 5.3 پیامدهای عملی

- پروژه انتظار سختگیری بیشتری نسبت به MVP معمولی دارد. مزیت آن کاهش technical debt بلندمدت و تطابق بهتر با نیازهای مالی/قانونی است.
- نقطه ضعف آن، تحویل اولیه کندتر است؛ تیم باید این سرعت محتاطانه را بپذیرد.

## 6. فرصت کسب‌وکار و احتمال موفقیت

### 6.1 چرا این محصول می‌تواند موفق شود

- مشکل واقعی و ملموسی را در بازاری آشنا هدف می‌گیرد.
- ترکیب ربات + PWA مشتری اصطلاحاً friction را کاهش می‌دهد و با رفتار کانالی ایران همخوانی دارد.
- قیمت‌گذاری و مدل درآمد tenant-paid با منطق SaaS B2B هم‌راستا است.
- پس از تثبیت پلتفرم، مسیر طبیعی توسعه به ماژول‌های مجاور وجود دارد.

### 6.2 عوامل کلیدی موفقیت

- پیاده‌سازی امن و صحیح multi-tenancy و RBAC از روز اول.
- یادآور قابل اعتماد و workflow واضح گزارش/تایید پرداخت.
- ورود pilot سریع با import Excel و فرایند لینک مشتری.
- case study واقعی از مشتریان اولیه.
- مستندسازی محکم و انضباط تسکی برای جلوگیری از بدهی معماری پنهان.

### 6.3 ریسک‌های اصلی شکست

- over-engineering فاز 0 و از دست دادن فرصت pilot.
- اشتباه در جداسازی tenant یا صحت جریان پرداخت.
- UX ضعیف یا نبود حالات خطا در UI فروشنده/مشتری.
- نداشتن آمادگی عملیاتی برای پایداری webhook ربات.

### 6.4 برآورد قابلیت موفقیت

- استراتژی مستند شده واقع‌بینانه و محتاطانه است و برای محصول سازمانی در این حوزه مناسب است.
- اگر تیم انضباط بیان‌شده را حفظ کند، احتمال پیروزی در pilot بالاست.
- این محصول کمتر مبتنی بر حدس و گمان است، زیرا درد و ارزش پیشنهادی قابل اندازه‌گیری دارد.

## 7. پیشنهادات برای فاز 0

### 7.1 حفظ تمرکز

- فاز 0 را به عنوان کار پایه‌ای نگه دارید و scope را بزرگ‌تر نکنید.
- صحت پلتفرم را بر روی زیبایی اضافی UI اولویت دهید.

### 7.2 اجرای قوانین

- قالب‌های مستندات و تسک را به دقت رعایت کنید.
- نیاز به traceability صریح بین docs → ADR → task → implementation باشد.

### 7.3 اعتبارسنجی زودهنگام

- یک vertical slice سریع اجرا کنید: ثبت tenant، ایجاد customer، ثبت فروش، یادآوری.
- تست‌های integration برای جداسازی cross-tenant و رفتار soft delete اجرا شوند.

### 7.4 نظارت بر حوزه‌های ریسک

- منطق session شعبه فعال و data scope را با سناریوهای واقعی نقش‌ها بررسی کنید.
- idempotency یادآورها و timezone تهران را تأیید کنید.
- جریان گزارش پرداخت و تأیید را شفاف بررسی کنید.
